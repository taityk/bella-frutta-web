# Web サイトプロジェクト — Bella Frutta Daikanyama

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| 目的 | 公式情報の一元化、検索・シェア時の信頼、Instagram からの離脱を減らす |
| 想定ユーザー | **トップ（`/`）は主に B2C**（来店前にメニュー・場所を確認したい人、SNS から流入）。**法人**は別ページ（`/corporate`）で案内し、トップからリンクで導線のみ置く。 |
| 成功指標 | 正確な情報での来店、問い合わせ削減、プロフィールリンク経由の滞在、**法人問い合わせの着手数**（`/corporate` のフォーム送信・メール） |

---

## 2. 方針：最初はコーディング、その後はスタッフが運用

### 2-1. フェーズのイメージ

| フェーズ | 目的 | 主な担当 |
|----------|------|----------|
| **A. 立ち上げ** | デザインどおりに動くサイトをコードで実装し、公開する | 実装者（あなた） |
| **B. 運用移行** | メニュー・営業時間・お知らせなどを **スタッフがブラウザから更新**できるようにする | 実装者が仕組みを用意 → スタッフが日常更新 |
| **C. 定常運用** | 月次の情報更新、写真差し替え、臨時休業の反映 | スタッフ（必要時のみ実装者へ依頼） |

「全部を最初から CMS に乗せる」より、**先に画面と情報設計をコードで固め、編集しやすい部分だけを CMS / データ化**すると、見た目のブレと学習コストのバランスが取りやすいです。

### 2-2. スタッフに任せたい更新の例

- メニュー名・価格・一言説明
- 営業時間・定休日（臨時休業）
- トップのお知らせ・キャッチの短文
- ギャラリー用の写真差し替え（枚数・比率はテンプレ固定が無難）
- **法人向け**の掲載例・注意書き（対応エリア・最低ロット等は法務・営業と整合）

**コードが必要になりやすい変更**（契約や見積で切り分け）:

- ページの増減、レイアウトの組み替え
- 新しいセクションのデザイン追加
- フォーム・外部サービス連携の変更（Google フォーム URL、メール送信バックエンド、CRM 連携など）

## 3. スコープ（MVP 推奨）

### 必須ページ

1. **トップ（`/`）** — **B2C 向け情報に限定**。一言キャッチ、代表ビジュアル、主要 CTA（メニュー・アクセス・LINE）。**法人向けは掲載せず**、専用ページへのリンクのみ（ヘッダー・帯・フッター等）
2. **法人のお客様（`/corporate`）** — 法人向けスムージー・アイス等のラインアップ概要、お問い合わせフォーム（または外部フォームリンク）
3. **メニュー（`/menu`）** — 商品名、価格、主要アレルゲン・サイズ（あれば）。**QRコードの飛び先として独立ページ化**（トップ内アンカーではなく `bellafutta.jp/menu` のような安定した URL）。1〜2週間に1回の更新頻度を想定し、フェーズ B で CMS 化を優先する
4. **果物カレンダー（`/calendar`）** — 旬の果物・産地の時期別一覧。季節の仕入れ先（あきやま農園のデコポン、出島の花のみかん等）を紹介するコンテンツページ。SEO 訴求にも活用
5. **アクセス** — 住所、地図埋め込み、最寄り駅、徒歩分、営業時間、定休日
6. **Instagram 連携** — 埋め込みまたは公式アカウントへの明確なリンク

### あれば良いページ

- **ストーリー / こだわり** — ブランド戦略（`01-brand-strategy.md`）と同じメッセージ
- **よくある質問** — 予約の要否、キャッシュレス、テイクアウト可否、**法人対応の可否・目安** など
- **プライバシーポリシー** — **法人向けお問い合わせフォーム**やメルマガを置く場合は必須（個人情報の利用目的の明示）

### 今回のスコープ外（初期）

- EC・定期購入（必要になった段階で拡張）
- 複雑な予約システム（LINE で代替可能なら後回し可）

### トップページのセクション構成（案）

ホームを **縦に読む一本のストーリー** にしつつ、メニュー・アクセスは **別 URL または同一ページ内アンカー** で辿れるようにすると、SNS 流入と SEO の両方に効きます。

| 順 | セクション | 内容 |
|----|-------------|------|
| 1 | **ヒーロー** | 外観・商品・果物などの写真を背景で大きく表示し、複数枚を切り替え（スライダー／フェード）＋キャッチコピー・主要 CTA |
| 2 | **お知らせ** | 新メニュー、臨時休業、キャンペーン、Instagram への導線など（運用フェーズ B 以降は CMS 更新を想定） |
| 3 | **果物へのこだわり** | 仕入れ・鮮度・選び方など。`01-brand-strategy.md` のメッセージと揃える。**水・氷ゼロ、果実100%** を明確に打ち出す。試食→購買→信頼のストーリーも組み込む。主要仕入れ先（あきやま農園・出島の花等）を短く紹介 |
| 4 | **作り手の思い** | 店・スタッフのストーリー。信頼と温度感（必須ページの「ストーリー／こだわり」に相当）。「飲む体験ではなく、五感で味わう体験」を言語化する |
| 5 | **商品一覧（B2C）** | 店頭・テイクアウト想定。**スムージー** / **カットフルーツ** 等。近い将来の展開ライン（アンチエイジング / 燃焼系 / リゾート系）は Coming Soon として示唆可能 |
| 6 | **法人導線（リンクのみ）** | 「法人のお客様はこちら」→ **`/corporate`**。トップ本体には法人向け長文・フォームを置かない |
| 7 | **店舗情報** | 住所、地図、営業時間、定休日、LINE 等（必須の「アクセス」と統合可能） |

**別ページ `/corporate`（法人向け）** にまとめる内容: 法人向けラインアップ（概要・用途例・「要相談」表記）、お問い合わせフォーム（または Google フォーム等へのリンク）、必要ならパンくずでトップへ戻す。

打ち合わせで判明した **追加の B2B 要件**（`/corporate` ページに反映する）:

| 用途 | 内容 |
|------|------|
| **ギフト** | 旬の果物を使ったギフト提案。贈り物シーンの訴求 |
| **旬の果物バルク** | 法人向けに旬の果物を安価・まとめて提供するサービスの示唆 |
| **OEM** | 有名美容クリニックが術後スムージーとして採用している実績（`[要確認]` クリニック固有名詞の掲載可否を店舗側に確認） |

**デザイン・構成の参考**: [サザエ食品株式会社 公式サイト](https://www.sazae.co.jp/) — 大きなビジュアルによるヒーロー、セクション見出しのリズム（日本語＋英字ラベル）、ラインナップ・店舗・ニュースの流れ、フッター付近の導線ブロックなど。Bella Frutta では業態に合わせ **お品書き相当＝商品一覧**、**読みもの／ジャーナルは初期は任意**（Instagram 中心なら後からでも可）と割り切ると MVP と矛盾しません。

## 5. 技術・実装オプション（コーディング起点＋スタッフ運用）

### 5-1. 方針別の組み合わせ

| 方針 | 構成の例 | スタッフ向け | 備考 |
|------|----------|--------------|------|
| **A. 静的サイト + Git 管理画面** | Astro / Next.js 等 + [Decap CMS](https://decapcms.org/)（旧 Netlify CMS） | ブラウザで Markdown やフォーム入力 → 承認後にデプロイ | リポジトリとホスティング（Netlify / Cloudflare Pages 等）の初期設定が必要。**更新は「プレビュー → 公開」の流れ**をマニュアル化 |
| **B. 静的/SSG + ヘッドレス CMS** | 同上 + [microCMS](https://microcms.io/) / [Newt](https://www.newt.so/) / Contentful 等 | 管理画面は日本語 UI が多く、フィールドがフォーム化されやすい | API 取得でビルド or ISR。**メニューを構造化データ**にするとスタッフが項目ごとに入力しやすい |
| **C. WordPress（ブロックテーマ）** | 最初から WP、または後から移行 | 慣れた人が多い。**固定ページ＋カスタム投稿タイプ「メニュー」** 等 | プラグイン・本体の更新運用が必要。セキュリティ対策（最低限の対策リストをマニュアルに） |
| **D. ハイブリッド** | トップ・ストーリーはコード固定、**メニュー・お知らせだけ CMS** | 更新頻度の高い部分だけ編集権を渡す | デザインの再現性が高い |

**推奨の検討順（コーディングできる前提）**

1. **更新の主役がメニュー・お知らせ** → **B（microCMS 等）+ SSG** または **A（Decap + Markdown/JSON）**
2. **スタッフに WordPress 慣れがある** → **C**
3. **とにかく月 1 回程度の軽い更新** → 最初は **JSON / YAML を GitHub の Web 編集**だけでも可（マニュアル必須）

> **⚠️ メニュー更新頻度について（2026年4月打ち合わせ確認）**: メニューは **1〜2週間に1回**変わる（産地が時期によって異なるため）。この頻度を考慮すると CMS の優先度は **高** に格上げ。フェーズ A でもメニューは `JSON` データ形式で管理し、GitHub の Web 編集で更新できる仕組みを推奨。フェーズ B では `/menu` ページを microCMS 等に最優先で接続する。

### 5-2. フェーズ A で決めておくこと（運用に効く）

- **コンテンツのデータ形**: メニューは「配列（商品ごとに名前・価格・説明・画像）」のように **型を決めておく**と CMS のフィールド設計が楽
- **画像**: 置き場所（CMS のアップロード vs クラウドストレージ）と **推奨サイズ・比率**
- **本番 URL・ステージング**の有無（スタッフが触る前に確認できる環境があると安心）
- **アカウント**: CMS は **店用メール**で登録し、**2 人まで**など編集者の範囲を決める

### 5-3. ノーコードのみの場合（参考）

Squarespace / Wix は最初からスタッフ運用向きだが、「最初にコーディングで作る」方針とは別ルート。移行する場合は **デザインの作り直し**が発生しやすい。

## 6. 多言語対応方針（日本語 ⇄ 英語）

> **店舗側の要望（2026年4月打ち合わせ）**: 日本語と英語で切り替えられるようにしたい。

### 6-1. 対応方針

- **フェーズ A に含める**（後回しにするとフェーズ B の CMS 移行時に大規模リファクタが必要になるため）
- ライブラリ: **`next-intl`** を導入（Next.js App Router と相性が良く、日本語ドキュメントも豊富）
- URL 構造: `/`（日本語デフォルト）、`/en/`（英語）
- 対象ページ: 全ページ（`/menu`・`/calendar`・`/corporate`・アクセスを含む）

### 6-2. コンテンツ準備

| 作業 | 担当 |
|------|------|
| 日本語テキストの英語翻訳（全掲載文） | 店舗側と協力して確認・承認 |
| メニュー名・仕入れ先紹介の英語版 | `[要確認]` 店舗側に英語表記の希望があれば提供してもらう |
| UI ラベル（ボタン・ナビ等） | 実装者が標準的な英語に翻訳 |

### 6-3. 実装上の注意

- テキストはすべて `messages/ja.json` / `messages/en.json` に抽出して管理する
- 画像に埋め込まれたテキストは切り替えが難しいため、CSSテキストに変更するか対応不要とする範囲を事前に確認する

## 7. 運用体制・引き継ぎ

### 7-1. 役割（例）

| 役割 | 担当例 | 内容 |
|------|--------|------|
| **オーナー / 責任者** | 店 | 掲載内容の最終確認、契約・アカウント名義 |
| **コンテンツ更新** | スタッフ | メニュー・時間・お知らせの CMS 操作 |
| **技術保守** | 実装者 or 月額サポート | 本体アップデート、不具合、レイアウト変更 |

### 7-2. 引き継ぎで渡すもの（チェックリスト）

- [ ] **運用マニュアル**（A4 で 2〜5 ページで可）: ログイン場所、メニューの直し方、画像の入れ方、公開の流れ、**やってはいけないこと**
- [ ] **アカウント一覧**（パスワードはセキュアな共有方法で）: ドメイン、ホスティング、CMS、Git（触る場合）
- [ ] **緊急時の連絡先**（サイトが落ちた・表示が崩れたとき）
- [ ] **月 1 回の定期作業**があれば明記（WordPress 更新など）

### 7-3. スタッフ向けのルール例

- 長文やキャンペーン文言は **一度下書きで保存**し、責任者確認後に公開
- 価格・アレルゲンは **紙メニューと必ず照合**
- 画像は **スマホの縦長だけでなく、指定の比率**でトリミング

### 7-4. 保守の契約パターン（目安）

| パターン | 内容 |
|----------|------|
| **スポット** | レイアウト変更・機能追加の都度見積 |
| **ライト月額** | 監視・軽微な修正・質問対応（月 ○ 時間まで） |
| **発生時のみ** | マニュアル外は都度依頼（非プロ運用と相性が良いことが多い） |

## 8. SEO・ローカル対策・SNS 整備

- タイトルに **「代官山」「スムージー」** 等を自然に含める（法人向けページがある場合は **「法人」「ギフト」「ケータリング」** など検索語を乱用せず事実ベースで）
- `h1` は店名＋業態が分かる一文
- **構造化データ**（LocalBusiness）は可能なら設定
- **Google ビジネスプロフィール** の住所・営業時間と完全一致
- CMS で本文を書き換えたあと **タイトル・ディスクリプション**が意図どおりか確認（テンプレで固定でも可）

### SNS・地図の整備（2026年4月打ち合わせで指摘された課題）

| 課題 | 対応 |
|------|------|
| **Google マップの情報整備** | Google ビジネスプロフィールの住所・営業時間・電話番号を Web サイトと完全一致させる。写真も追加して視認性を上げる |
| **Instagram の地図に出てこない問題** | Instagram プロフィールの住所設定を確認・有効化する。位置情報タグの活用も推奨 |
| **Facebook** | Facebook ページの有無を確認。存在する場合は Web サイトからリンクを追加（`web/src/lib/site-links.ts` に追記）。不要と判断した場合は明記してスコープ外にする |
